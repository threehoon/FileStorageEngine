# 分布式核心存储引擎

### 项目背景

截至2016年，淘宝平台的卖家数量已超过900万，商品总量突破10亿。每件商品包含大量图片与文字，平均数据量约为15KB。粗略估算，整个系统的存储需求已超过1PB。若使用单块1TB硬盘存储，至少需要**1024块**硬盘才能满足需求。

 **1PB = 1024TB = 1024 × 1024GB**。

如此庞大的数据量显然无法依赖单个文件或单台服务器进行存储和管理。

为应对海量的非结构化数据存储需求，淘宝自主研发了 **TFS（Taobao File System）**，一种基于 Linux 服务器集群的分布式存储系统。

TFS 具备 **高可靠性、高并发访问** 等特性，为淘宝平台提供稳定高效的数据存储解决方案。

### 项目介绍

( 运用大文件进行存储 一般为64M~128M存储非结构优化数据 ) 其中主要包含索引文件、主块文件、扩展块，索引文件(主要运用hash表的数据结构来存储文件的位置、大小等信息)，通过一个脚本把索引文件mmap到 内存，再通过每个hash表中的每一个metaInf o结点中记录的文件在主块的偏移，文件大小进行快速查找；主块文件则是实例存储数据的地方；扩展快存储超出主块容量的数据。

### 需要了解的一些概念

#### 1、什么是文件系统？

将我们的数据整合成目录或者文件，提供对文件的存取接口，基于文件的权限进行访问。简单的说，文件系统就是对文件进行管理的方式 。

![文件系统](img\文件系统.png)

#### 2、文件存储的一些概念？

![磁盘](img\磁盘.png)

**扇区** :  存储数据的磁盘的最小单位 , 通常一个扇区的大小是 512B( 相当于0.5kB )  == B(字节)。 

**块**: 文件存储的基本单位, 一般是连续的**八**个扇区组成一块 , 512B * 8  == 4096B  == 4kB。

#### 3、文件的结构？

**文件 **== 目录项  +  块数据项 +  Inode 元信息。

**目录区 **: 存放目录下文件的信息 。

**块数据区** : 存储文件数据 。

**Inode元信息区 **: 存放文件的 `inode` 包含的信息 ,我们可以理解为文件的属性 。

**Inode 结点 : **  可以用,命令 `ls -il` 来查看一下。

![inode信息](img\inode信息.png)

**索引结点 **: 用来存取文件的元信息, 包括文件大小,创建者 ,创建日期等等 。每一个文件都有一个inode号码,相当于我们的身份证,操作系统操作系统根据 inode 编码来识别文件 。

**结点大小:**一般是128和256个字节 ， inode结点的数量在最开始格式化磁盘的时候就已经标记 , 相当于一个组成一个*inode table* , 一般是每1kB 或者 2kB 设置一个结点。例如 1GB的硬盘 , 按128 B和 1kB来算 , *inode*结点大概会占用128MB大小的空间, 占整个磁盘的12.8% 。 所有如果你发现你的磁盘还剩10个G的内存,但是死活存不进数据 ，是因为被我们的*inode*结点占据了 。

#### 5、为什么采用大文件(block file)结构的原因？

- 小文件存储海量的数据，会导致我们硬件的存储器的磁头需要频繁的更换磁道，就会提高我们的存取时的时间(延时)。
- 如果进行频繁的增加和删除，会导致我们的磁盘产生很多碎片，降低磁盘的存取效率，同时也降低了IO设备的读写效率。
- 海量的小文件，会导致*inode* 编号占用太多磁盘空间，降低缓存的效果。

#### 6、文件映射

磁盘的速度，和内存的速度比较，相当于是 ： 一个走路,一个坐火箭。

文件映射（Memory-Mapped File, MMF）是一种将**文件的部分或全部内容映射到进程的虚拟地址空间**的机制。它允许程序直接像操作内存一样读写文件，而不需要调用 `read()` 或 `write()` 进行 I/O 访问。

##### 优点:

1、避免 `read()` 和 `write()` 系统调用的开销。
2、数据直接在用户空间和磁盘之间映射，无需拷贝到内核缓冲区。
3、多个进程可以映射同一文件，实现**进程间通信（IPC）**。
4、数据只有在访问时才从磁盘加载到内存，减少 I/O 操作。

##### `mmap()` 系统调用

```cpp
void* mmap(void* addr, size_t length, int prot, int flags, int fd, off_t offset);
参数	   说明
addr	指定映射的起始地址，通常为 nullptr（让内核自动分配）
length	映射的字节数
prot	访问权限：PROT_READ（可读）、PROT_WRITE（可写）
flags	MAP_SHARED（共享映射）或 MAP_PRIVATE（私有映射）
fd	    文件描述符
offset	文件内的偏移量，必须是 页面大小的整数倍（通常 4KB）
```

##### **`msync()` 同步映射**

修改后的数据不会立即写回磁盘，可用 `msync()` 强制同步：

```cpp
int msync(void* addr, size_t length, int flags);
参数	   说明
addr	内存映射的起始地址
length	需要同步的范围
flags	MS_SYNC（同步写入）、MS_ASYNC（异步写入）
```

##### `munmap()` 解除映射

```cpp
int munmap(void* addr, size_t length);
//  映射完成后，调用 munmap() 释放映射区域：
```

##### 内存映射的流程：

```cpp
// 1、打开文件 (  open()  )：
以 O_RDWR 或 O_RDONLY 方式打开文件，获得文件描述符（fd）。
//2、创建映射 (  mmap()  )：
调用 mmap()，将文件的部分或全部内容映射到虚拟地址空间。
//3、访问映射数据：
直接访问映射的地址，就像操作数组一样，无需 read()。
//4、同步更新（可选） (  msync()  )：
修改的数据不会立即写回磁盘，可使用 msync() 强制同步。
//5、解除映射 (  munmap()  )：
使用 munmap() 释放映射的内存区域。
```

```cpp
//  demo:
#include <iostream>
#include <fcntl.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <unistd.h>
#include <cstring>

int main() {
    const char* filename = "example.txt";

    // 1. 打开文件
    int fd = open(filename, O_RDWR | O_CREAT, 0666);
    if (fd < 0) {
        perror("open");
        return 1;
    }

    // 2. 获取文件大小
    struct stat sb;
    if (fstat(fd, &sb) == -1) {
        perror("fstat");
        return 1;
    }
    size_t filesize = sb.st_size;

    // 3. 扩展文件大小（如果文件为空）
    if (filesize == 0) {
        ftruncate(fd, 1024); // 扩展到1KB
        filesize = 1024;
    }

    // 4. 创建文件映射
    void* map = mmap(nullptr, filesize, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);
    if (map == MAP_FAILED) {
        perror("mmap");
        return 1;
    }

    // 5. 读取和修改文件数据
    char* data = static_cast<char*>(map);
    std::cout << "原始数据: " << data << std::endl;

    strcpy(data, "Hello, Memory Mapped File!"); // 直接修改文件内容

    // 6. 同步到磁盘
    msync(map, filesize, MS_SYNC);

    // 7. 解除映射并关闭文件
    munmap(map, filesize);
    close(fd);

    return 0;
}

```

### 设计思路

1、数据文件以 **块（block）** 形式存储，每个块通常为 **64MB**，并具有唯一的整数编号。在使用之前，块所占用的存储空间都会经过**预分配和初始化**，确保可用性和一致性。

2、每个数据块由一个索引文件、一个主块文件以及若干个扩展块组成。主块主要用于存储“小文件”，而扩展块则专门存储溢出数据。

3、索引文件用于存储对应块的信息及“小文件”的索引数据，并在服务启动时通过 *mmap* 映射到内存，以大幅提升文件检索速度。“小文件”索引采用哈希链表结构在索引文件中实现。

4、每个文件都有唯一的文件编号，从 1 开始递增，同时作为哈希查找的 Key，用于定位“小文件”在主块和扩展块中的偏移量。通过特定算法，文件编号与块编号可生成对应的“小文件”文件名。

#### 哈希链表

哈希链表是一种结合**哈希表**和**链表**的数据结构，主要用于高效索引和存储多个元素，避免哈希冲突。它在大文件存储索引中可用于管理“小文件”的偏移量，提高检索速度。典型的“空间换时间”的做法。

在这个项目中呢。

哈希链表的

**值(value)**: 文件的索引信息(  文件大小 、位置  )。

**键（key）**: 文件编号。如1，2，3，4；

**哈希桶: **Hash 表中存放数据的基本存储单元，一个桶可以存放 0~多个元素，多个桶组成 Hash 表,比如说下图0、1、2都是一个hash桶。

**Hash函数**：key % 5。

### 具体流程

#### 一 :  块初始化

#####  **1. 生成主块文件**

- 根据 `block_id_` 创建文件（如 `block_1.data`）。
- 预分配空间（ 使用 `ftruncate()` 预留磁盘空间，提高写入性能 ）。

#####   2.生成索引文件

- 创建索引文件（如 `index_1.meta`）。

- 初始化文件头部（ 存储元信息，如哈希表大小 ）。

- 同步写入磁盘（ 避免断电丢失 ）。

- 映射至内存访问（ `mmap()` 提高访问效率  ）。

#### 二：块中写入文件

**1. 加载索引文件**：

- 将索引文件映射至内存，以便快速访问。

**2. 文件写入至主块**：

- 从索引文件读取块的数据偏移量。
- 将文件数据写入主块的正确位置。

**3. 索引信息写入索引文件**：

- 生成 `MetaInfo` 结构，存储文件位置信息。
- 写入索引文件，并更新 `BlockInfo`（管理块内的文件信息）。

#### 三：块中读取文件

##### 1. 加载索引文件：

- 使用 `mmap()` 将索引文件映射到内存，以便高效访问 `MetaInfo` 结构。

##### 2. 获取文件 `MetaInfo`：

- 根据 `fileid` 在索引表中查找 `MetaInfo`。

##### 3. 读取文件数据：

- 根据 `MetaInfo` 解析文件偏移量 ( `inner_offset` )。
- 使用 `pread()` 直接从 `Main Block` 读取文件内容。

#### 四： 块中删除文件

##### 1. 加载索引文件

- 使用 `mmap()` 将索引文件映射到内存。

##### 2. 获取文件 `MetaInfo`

- 基于哈希索引 在 `MetaInfo` 链表中找到对应 `fileid` 。
- 如果发生哈希冲突，通过 `next_meta_offset` 遍历链表。

##### 3. 删除 `MetaInfo`

- 调整链表指针：让 前一个 `MetaInfo` 连接到后一个 `MetaInfo`，从而移除当前 `MetaInfo`。
- 回收 `MetaInfo` 空间
- 维护空闲链表，存储已删除的 `MetaInfo` 位置，便于后续重用。
- 更新 `BlockInfo`
- 减少 `file_count`，增加 `del_file_count` 和 `del_size` 记录删除状态。

### 高频面试问题

#### 1、项目中的索引机制是如何设计的？为什么选择哈希索引？

* 哈希索引 + 链表，解决冲突问题，提供高效的小文件查找能力。

* 索引文件 mmap 到内存，避免 read/write 造成的内核态与用户态数据拷贝，提高访问速度。

* 文件编号（fileid）+ Hash 函数映射到 Hash Bucket，实现 O(1) 级别的查找速度。

#### 2、为什么采用 mmap 进行索引管理？相比传统文件读写方式有什么优势？

**传统方式（read/write）**：需要多次系统调用，涉及内核态与用户态数据拷贝，性能开销较大。

**mmap 方式**：

- 直接将文件映射到进程地址空间，减少数据拷贝。
- 只在访问时加载，不占用额外内存，提升 I/O 速度。
- 适用于高并发读写场景，如索引文件查询。

#### 3、文件写入流程是怎样的？如何保证数据一致性？

* 文件编号计算（fileid → Hash 桶）。

* 查找可用数据块（block）。

* 找到 block 的空闲区域（主块 or 扩展块）。

* 写入数据到主块，并更新索引文件（文件偏移 + 大小）。

* 使用 msync() 或 `fdatasync()`保证数据写入磁盘，防止断电丢失。

#### 4、如何处理文件删除？会产生什么问题？如何优化？

* 删除文件时，不会立即删除数据，而是标记为删除状态。

* 删除的 `MetaInfo` 需要维护空闲链表，避免产生索引碎片。

**垃圾回收机制**：

- 通过 后台合并机制定期回收主块中过多的小文件碎片。
- 如果主块碎片过多，可以 新建数据块并迁移活跃数据。

#### 5、如果索引文件 mmap 失败了怎么办？

mmap 失败的原因：

- 文件过大导致 地址空间不足。
- 物理内存不足，导致无法分配连续虚拟地址。

解决方案：

- 采用分段 mmap（仅加载部分索引，提高内存利用率）。
- 减少 mmap 文件大，调整索引存储策略。
- 增加交换分区（swap），缓解内存不足问题。

#### 6、你在这个项目中遇到的最大挑战是什么？如何解决的？

问题：索引文件 mmap 时，文件过大会导致内存不足。

优化方案：采用 分块 mmap，只映射部分索引，提高内存利用。

#### 7、与传统文件系统（如 ext4、XFS）的区别是什么？

传统文件系统：

- 适用于 通用存储，对小文件存储支持较差（inode & block 结构限制）。
- 文件元数据（inode）管理复杂，小文件过多时会导致 磁盘碎片化。

本存储方案：

- 面向小文件优化，通过 索引+数据块结构 降低碎片率，提高查找速度。
- 顺序写入 + 预分配数据块，减少磁盘 seek 开销。
- mmap 读写索引，提高小文件查找速度。

#### 8、为什么 采用大文件存储（Block File），而不是直接存储小文件？

直接存储小文件会导致：

- 磁盘 IOPS 开销大（文件过多时，创建/删除操作开销大）。
- 文件系统 inode 资源耗尽，导致存储效率下降。

**TFS 采用 Block File**：

- 小文件打包成大文件，减少 inode 元数据管理开销。
- 索引管理（哈希索引）+ mmap 提升查询速度。
- 顺序写入块文件，提高磁盘利用率，减少随机写入。

#### 9、如何解决hash冲突的？

采用 链地址法（哈希链表），避免哈希碰撞导致的查询失败。

#### 10、如何保证数据的一致性？如果发生宕机，如何恢复数据？

* msync() / fdatasync() 确保 mmap 数据写入磁盘，防止因进程崩溃导致索引丢失。

* 索引文件映射到内存，定期 `fsync()` 刷盘，防止索引丢失。

* 主块文件不随意删除，只在合适时机清理过期文件，防止数据丢失。

#### 11、如何支持大文件存储？超过 block 限制怎么办？

扩展块（Extend Block）机制：如果主块存储满了，创建扩展块存储额外数据。

数据分片：将大文件拆分成多个 block 存储，提高存取效率。
